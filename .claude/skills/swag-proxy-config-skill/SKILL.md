# SWAG Proxy Config Generator

Generate secure nginx proxy configurations for LinuxServer.io SWAG with Authentik forward auth, hardened security headers, rate limiting, and fail2ban integration.

## Repo-Specific Context

This repo uses LSIO SWAG with template-based Authentik integration. The key files:

- **Source configs** live in `services/vps/` (edit these, never generated files directly)
- **Generated configs** end up in `docker/swag/config/nginx/proxy-confs/` via `build-proxy-confs.sh`
- **Authentik includes** (`authentik-server.conf`, `authentik-location.conf`) are generated by `build-proxy-confs.sh` from templates in `docker/swag/nginx/`
- **Naming**: `<service>.subdomain.conf` using `server_name <service>.*;` (wildcard matches all domains)

## Why This Skill Exists

SWAG proxy configs combine several independent security mechanisms that must be correctly wired together. The Authentik forward auth integration involves multiple nginx locations with specific header propagation. Getting `auth_request` wrong — or placing it on the wrong location block — creates a false sense of security where the auth gate appears to work but can be bypassed. This skill produces complete, correct configs with all security layers properly integrated.

## Config Template: With Authentik (Standard — Most Services)

```nginx
# <Service Name> - VPS Docker container
# Upstream: <container-name>:<port> (Docker container on proxy network)
# Auth: Authentik forward auth (enabled)

server {
    listen 443 ssl;
    listen [::]:443 ssl;
    http2 on;

    server_name <service>.*;

    include /config/nginx/ssl.conf;
    include /config/nginx/proxy.conf;

    # Authentik auth endpoint
    include /config/nginx/authentik-server.conf;

    location / {
        # Require Authentik SSO before access
        include /config/nginx/authentik-location.conf;

        include /config/nginx/resolver.conf;
        set $upstream_app <container-name>;
        set $upstream_port <port>;
        set $upstream_proto http;
        proxy_pass $upstream_proto://$upstream_app:$upstream_port;
    }
}
```

## Config Template: With WebSocket Support

For services using Socket.IO, WebSocket, or SSE (Uptime Kuma, n8n, code-server):

```nginx
    location / {
        include /config/nginx/authentik-location.conf;

        include /config/nginx/resolver.conf;
        set $upstream_app <container-name>;
        set $upstream_port <port>;
        set $upstream_proto http;
        proxy_pass $upstream_proto://$upstream_app:$upstream_port;

        # WebSocket / Socket.IO support
        proxy_http_version 1.1;
        proxy_set_header Upgrade $http_upgrade;
        proxy_set_header Connection $connection_upgrade;
    }
```

Services that need websockets: Uptime Kuma (Socket.IO), Authentik, Homarr, n8n, code-server.

## Config Template: Without Authentik (Own Auth / Public)

For services with their own sufficient auth (Vaultwarden, Plex, Jellyseerr):

```nginx
# <Service Name> - VPS Docker container
# Upstream: <container-name>:<port>
# Auth: None - <service> handles its own authentication

server {
    listen 443 ssl;
    listen [::]:443 ssl;

    server_name <service>.*;

    include /config/nginx/proxy.conf;

    location / {
        include /config/nginx/resolver.conf;
        set $upstream_app <container-name>;
        set $upstream_port <port>;
        set $upstream_proto http;
        proxy_pass $upstream_proto://$upstream_app:$upstream_port;
    }
}
```

## Multiple Location Blocks

When a service has API endpoints that need separate handling, **every location that serves content must have its own `auth_request`** via `authentik-location.conf`:

```nginx
    location / {
        include /config/nginx/authentik-location.conf;
        include /config/nginx/resolver.conf;
        set $upstream_app <service>;
        set $upstream_port <port>;
        set $upstream_proto http;
        proxy_pass $upstream_proto://$upstream_app:$upstream_port;
    }

    location ~ (/service)?/api {
        # Auth required on API paths too!
        include /config/nginx/authentik-location.conf;
        include /config/nginx/resolver.conf;
        set $upstream_app <service>;
        set $upstream_port <port>;
        set $upstream_proto http;
        proxy_pass $upstream_proto://$upstream_app:$upstream_port;
    }
```

## Rate Limiting

Rate limit zones are defined globally in SWAG's nginx.conf. Apply per-location:

```nginx
    # For authentication endpoints (strict)
    location /api/auth {
        limit_req zone=auth_limit burst=3 nodelay;
        include /config/nginx/authentik-location.conf;
        # ... proxy config
    }
```

Common zones (defined in SWAG's global nginx.conf):
- `auth_limit` — 5 req/min, for login endpoints
- `api_limit` — 30 req/min, for API endpoints
- `general_limit` — 60 req/min, for normal browsing

**Rate limiting goes BEFORE `proxy_pass`** — placing it after means the request already hit upstream.

## Fail2ban Integration

### Filter (`fail2ban/filter.d/<service>.conf`)

```ini
[Definition]
failregex = ^<HOST> .* "(GET|POST) /(login|api/auth).* HTTP/.*" (401|403) .*$
ignoreregex =
```

### Jail (`fail2ban/jail.d/<service>.conf`)

```ini
[<service>]
enabled  = true
port     = http,https
filter   = <service>
logpath  = /config/log/nginx/access.log
maxretry = 5
findtime = 600
bantime  = 3600
action   = %(action_)s
```

## Rules (always follow these)

1. **Authentik forward auth must cover ALL locations that serve content.** If you have `location /api/` and `location /` blocks, BOTH need `include /config/nginx/authentik-location.conf;`. A missing include means that path bypasses Authentik entirely.

2. **Never put `authentik-location.conf` inside the Authentik outpost location.** The `authentik-server.conf` include handles that endpoint — gating it behind itself causes infinite redirects.

3. **Use `server_name <service>.*;`** (wildcard) — this is the repo pattern. It matches `service.dagint.com`, `service.wopr.net`, etc. with a single wildcard cert.

4. **Always include `http2 on;`** on the listen lines — the repo pattern requires it.

5. **Use the `$upstream_app` / `$upstream_port` / `$upstream_proto` pattern** with `resolver.conf` — this is how LSIO SWAG handles dynamic upstream resolution to avoid nginx startup failures when containers aren't running.

6. **Edit `services/vps/` files, not `docker/swag/config/nginx/proxy-confs/` directly.** The proxy-confs directory is generated output — direct edits get overwritten by `build-proxy-confs.sh`.

7. **Test after every config change.** After modifying a proxy config and running `build-proxy-confs.sh`, validate with `docker exec swag nginx -t` and check `docker logs swag` for errors. A syntax error in any config file takes down ALL proxied sites.

8. **`proxy.conf` includes `proxy_buffering off` and standard proxy headers.** Don't repeat those manually unless you need to override something specific.

## Checklist (verify before delivering)

- [ ] File placed in `services/vps/` (not in proxy-confs directly)
- [ ] Filename is `<service>.subdomain.conf`
- [ ] `server_name <service>.*;` (wildcard, with `http2 on;`)
- [ ] `include /config/nginx/ssl.conf;` and `include /config/nginx/proxy.conf;` present
- [ ] `include /config/nginx/authentik-server.conf;` in server block (if auth enabled)
- [ ] `include /config/nginx/authentik-location.conf;` in EVERY content-serving location (if auth enabled)
- [ ] Uses `set $upstream_app` / `set $upstream_port` / `set $upstream_proto` + `resolver.conf`
- [ ] WebSocket headers present if service uses Socket.IO/WebSocket
- [ ] Rate limiting applied to auth endpoints (if applicable)
- [ ] `build-proxy-confs.sh` run + `nginx -t` passes before commit
